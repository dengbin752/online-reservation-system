# 希尔顿餐厅在线预订系统 - 介绍演讲稿

---

## 一、项目概述

尊敬的面试官，您好！我今天要介绍的是我独立开发的**希尔顿餐厅在线预订系统**。这是一个面向高端餐饮行业的全栈预订管理平台，旨在为希尔顿旗下餐厅提供一套高效、便捷的数字化预订解决方案。

### 核心价值
- **客户侧**：顾客可以在线完成注册、登录、创建预订、查看个人预订历史、取消预订等操作
- **管理侧**：餐厅员工可以实时查看所有预订、审核新预订（确认/拒绝）、按日期和状态筛选订单
- **系统侧**：实现完整的认证授权机制，支持基于角色的访问控制

## 二、业务功能详解

### 2.1 客户功能模块

| 功能 | 描述 |
|------|------|
| **用户注册/登录** | 支持邮箱注册，JWT token 认证登录，支持 token 刷新机制 |
| **创建预订** | 填写联系方式、用餐人数、期望日期时间、特殊需求，提交预订申请 |
| **我的预订** | 查看个人所有预订记录，包括待确认、已确认、已完成等状态 |
| **取消预订** | 用户可自主取消尚未完成的预订 |

### 2.2 员工管理模块

| 功能 | 描述 |
|------|------|
| **预订仪表盘** | 展示所有客户提交的预订列表 |
| **预订审核** | 对新提交的"待确认"预订进行确认或拒绝操作 |
| **状态管理** | 支持将预订状态变更为：待确认(PENDING)、已确认(CONFIRMED)、已取消(CANCELLED)、已完成(COMPLETED)、未到(NO_SHOW) |
| **筛选查询** | 支持按日期、状态筛选预订记录 |
| **详情查看** | 查看每位顾客的联系方式、特殊需求等详细信息 |

### 2.3 预订数据模型

```
预订记录包含以下核心字段：
- customerId: 顾客ID
- tableId: 桌台ID
- date: 用餐日期
- time: 用餐时间
- partySize: 用餐人数
- status: 预订状态
- specialRequests: 特殊要求（如：靠窗座位、素食等）
- createdAt/updatedAt: 创建和更新时间戳
```

## 四、核心特性与亮点

### 4.1 混合 API 设计
- **RESTful API**：用于用户认证（登录、注册、token 刷新、登出）
- **GraphQL API**：用于业务逻辑（预订管理、查询、状态更新），支持灵活的数据获取和批量操作

### 4.2 完善的认证授权
- JWT Token 双 token 机制（access token + refresh token）
- 基于角色的访问控制（RBAC）：普通顾客 vs 餐厅员工
- Token 过期自动跳转登录页

### 4.3 数据持久化
- 使用 Couchbase NoSQL 数据库
- 实现了 Repository 模式封装数据操作
- 支持预订状态变更的历史追踪（version 字段）

### 4.4 前端交互体验
- 使用 SolidJS 实现响应式 UI
- 支持预订状态的实时可视化（颜色标签）
- 员工端支持模态框查看详情、一键确认/拒绝
- 表单验证与错误提示

### 4.5 开发体验
- 采用 pnpm workspace 实现 Monorepo 管理
- 共享包统一维护类型定义，避免重复代码
- Docker 容器化部署，支持一键启动
